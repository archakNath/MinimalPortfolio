When working with Spring Boot applications, security is not just an add-on—it’s a core requirement. One of the most important yet often misunderstood concepts in Spring Security is the **Security Filter Chain**.

If you’ve ever configured authentication or JWT-based security and wondered *“what actually happens behind the scenes?”*, this blog will give you a clear and practical understanding.

***

## What is the Spring Security Filter Chain?

At its core, the **Spring Security Filter Chain** is a sequence of filters that intercept every incoming HTTP request before it reaches your application’s controller.

Think of it like a **security checkpoint pipeline**:

> Request → Filters → Controller → Response

Each filter performs a specific task such as:

* Authentication
* Authorization
* CSRF protection
* Session management
* Exception handling

***

## Why is it Important?

Without the filter chain:

* Your endpoints would be publicly accessible
* No authentication checks would happen
* No role-based access control

In short, **your application would be completely insecure**.

***

## How the Filter Chain Works

When a request is sent to your application:

1. It first hits the **Servlet container**
2. Then passes through the **Spring Security Filter Chain**
3. Each filter decides:
   * Continue processing
   * Modify request/response
   * Reject the request

If all checks pass, the request reaches the controller.

***

## Key Filters in the Chain

Here are some commonly used filters:

### 1. UsernamePasswordAuthenticationFilter

* Handles login requests
* Extracts username and password
* Delegates authentication to AuthenticationManager

### 2. BasicAuthenticationFilter

* Handles HTTP Basic authentication
* Reads credentials from headers

### 3. BearerTokenAuthenticationFilter (JWT)

* Used in modern APIs
* Extracts JWT token from Authorization header

### 4. SecurityContextPersistenceFilter

* Stores authentication details in SecurityContext

### 5. ExceptionTranslationFilter

* Handles authentication/authorization errors

***

## Custom Filter Example (JWT)

In real-world applications, especially microservices, we often use JWT authentication.

Here’s how a custom filter fits into the chain:

```java
@Component 
public class JwtAuthFilter extends OncePerRequestFilter { 
  
  @Override 
  protected void doFilterInternal(HttpServletRequest request, 
                                  HttpServletResponse response, 
                                  FilterChain filterChain) throws ServletException, IOException { 
    String authHeader = request.getHeader("Authorization"); 
    if (authHeader == null || !authHeader.startsWith("Bearer ")) { 
      filterChain.doFilter(request, response); 
      return; 
    } 
    String token = authHeader.substring(7); // Validate token (pseudo logic) 
    if (isValid(token)) { 
      UsernamePasswordAuthenticationToken authToken = new UsernamePasswordAuthenticationToken(user, null, authorities); 
      SecurityContextHolder.getContext().setAuthentication(authToken); 
    } 
    filterChain.doFilter(request, response); 
  } 
}
```

## Configuring the Filter Chain (Spring Boot 3+)

With modern Spring Security, we define a `SecurityFilterChain` bean:

```java
@Bean 
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception { 
  return http 
    .csrf(csrf -> csrf.disable()) 
    .authorizeHttpRequests(auth -> auth
                           .requestMatchers("/api/auth/**")
                           .permitAll() 
                           .anyRequest()
                           .authenticated()
                          ) 
    .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class) 
    .build(); 
}
```

Key Points:

* `permitAll()` → Public endpoints
* `authenticated()` → Secured endpoints
* `addFilterBefore()` → Inject custom filter in the chain

***

## Filter Order Matters

Filters execute in a **specific order**, and placing your custom filter incorrectly can break authentication.

Example:

* JWT filter should run **before** `UsernamePasswordAuthenticationFilter`
* Otherwise, authentication won’t be set properly

***

## Common Mistakes

1. **Not setting authentication in SecurityContext**
   * Leads to 403 errors even with valid tokens
2. **Wrong filter placement**
   * JWT filter added too late in the chain
3. **Forgetting to disable CSRF for APIs**
   * Causes unexpected 403 errors
4. **Not handling null or invalid tokens**
   * Can break request flow

***

## Real-World Flow (JWT-Based API)

1. Client sends request with JWT token
2. JWT Filter extracts token
3. Token is validated
4. Authentication is set in SecurityContext
5. Request proceeds to controller
6. Controller returns response
